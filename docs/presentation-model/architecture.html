JSON Forms 4.x — Presentation Model Architecture

Draft Proposal  Targets eclipsesource/jsonforms#2571  ·  Branch feat/presentation-model  ·  June 2026

TL;DR — JSON Forms 4.x replaces the per-renderer mapStateTo*Props mapper layer with a centralized, framework-agnostic Presentation Model (PM): a serializable collection of nodes, built by a pluggable pipeline in @jsonforms/core, where every node carries everything a renderer needs — final label, translated errors, value, visibility, enablement, options. Renderers become thin views. The pipeline abstracts over the schema format (JSON Schema first, zod / arbitrary definitions later) and over validation (AJV first, anything later), and the model can be built on a server with no form engine running in the browser at all.

Status update (June 2026) — implementation has started as a clean cut: the repository was reset and rebuilt around the new architecture (pnpm + Turborepo + tsdown, ESM-only, React 19, MUI v9). Decisions taken since this document was first written:

  • No escape hatches. The originally proposed NodeSource raw-schema bag on nodes was dropped — nodes expose abstracted facets only.
  • Package renames: material-renderers → react-material, vanilla-renderers → react-vanilla; the example packages are replaced by apps/demo.
  • New package @jsonforms/validator-ajv: core has zero dependencies; AJV ships as a separate FormValidator implementation. Its base entry takes a caller-supplied AJV instance (types-only import; includes a precompiled-validator wrapper for no-eval environments), while per-draft presets live in subpath exports (/draft-07, /draft-2019, /draft-2020) so consumers bundle exactly one AJV build.
  • Issue display is a builder policy (IssueDisplayPolicy, default: show once touched) — renderers display node.issues unconditionally instead of combining interaction state themselves.
  • Boolean node flags are optional and default-false (hidden, disabled, required, readonly, touched) — builders only set what applies; serialized models stay lean.
  • Commands carry optional sourceNodeId provenance; React renderers dispatch through the node-scoped useControlDispatch and never see paths or ids.

1. Goals & non-goals

Goals

  • Single source of truth for presentation logic. Labels, descriptions, visibility, enablement, required-ness, error messages, enum options, i18n — computed once, centrally, in @jsonforms/core. Renderers only render.
  • Schema-format independence. JSON Schema is the first SchemaSource, not a hard-wired assumption. Other dialects, zod, or fully arbitrary definitions can be plugged in without touching renderers.
  • Validation independence. AJV is the first Validator, not part of the core state. Environments that cannot run AJV (CSP without unsafe-eval, restricted runtimes, bundle-size constraints) can use precompiled validators, custom lightweight validators, or server-side validation.
  • Location independence. The model is plain serializable data and the builder has no DOM/browser dependency — it can run on a server. A browser client can consume a server-built model with no form engine shipped to the client at all.
  • Performance by design. Updates preserve object identity for unchanged nodes and (in a later phase) recompute only affected nodes, instead of re-running all mappers for all renderers on every change.
  • Cross-cutting customization. Change label computation (or any other derived aspect) once via the pipeline, instead of overriding every renderer.

Non-goals (for the MVP)

  • No backward-compatible mapper/tester shims in the first iteration. The 3.x stack stays untouched and fully functional next to the new code; a compat layer is a later, separate decision (see roadmap).
  • No second schema source implementation in the MVP — but every interface is designed and reviewed against the question “would this work for zod / a plain definition / a server DTO?”.
  • No Angular/Vue bindings in the MVP; React proves the binding model first. The PM itself is framework-agnostic, so the other bindings follow the same pattern later.

2. Current architecture & pain points

Today, core keeps a minimal central state and every renderer derives its own props on every render through the mapper layer:

Concrete observations from the codebase that motivate the redesign:

Pain pointWhere it lives today
Presentation logic leaks into the “UI” layer: label derivation, required handling, error extraction, i18n key resolution, option computation, visibility rules. core/src/mappers/renderer.ts (~1,418 LOC) + per-renderer logic such as merge({}, config, uischema.options) in MaterialInputControl.
Customization is coarse-grained — changing how labels are computed means overriding every renderer. No central hook; the mappers are plain functions baked into HOCs.
AJV is structurally embedded: ajv and validator are fields of JsonFormsCore; AJV's ErrorObject is the public error type everywhere. core/src/reducers/core.ts, core/src/store/type.ts, core/src/util/errors.ts.
Every change re-runs all mappers and all testers; testers resolve schemas at match time. Memoization is bolted on per-HOC and is fragile (e.g. i18n function identity invalidates option memos). react/src/JsonFormsContext.tsx, react/src/JsonForms.tsx (TestAndRender).
Schema resolution ($ref, combinators) happens repeatedly at render time instead of once at build time. core/src/util/resolvers.ts called from mappers and testers.

3. Target architecture overview

The new architecture introduces three core concepts in @jsonforms/core:

  1. The Presentation Model — a versioned, serializable collection of presentation nodes mirroring the rendered form (§4).
  2. The builder pipeline — pure build logic composed from pluggable services: SchemaSource, Validator, ConditionEvaluator, i18n, and user-supplied NodeProcessors (§5).
  3. The FormEngine — the stateful runtime that owns data + model, accepts commands, and emits model deltas (§6).

Hard design rules for everything inside the presentation layer:

  1. Nodes are plain serializable JSON. No functions, no class instances, no live references into a schema object graph. This is what makes server-side building, dev tooling, snapshot testing, and AI-agent inspection possible.
  2. Nodes are complete. A renderer must never need to reach back into schema, data, or i18n state. If a renderer needs it, it goes on the node (or the node is extended via a NodeProcessor).
  3. Strings are final. Labels, descriptions, error messages are translated during the build. Locale switches trigger a rebuild (rare, acceptable).
  4. Ids are stable, identity is preserved. Unchanged nodes keep the same object reference across updates so view-layer memoization works for free.
  5. The pipeline is pure. Same inputs → same model. All statefulness lives in the FormEngine.

4. The Presentation Model

4.1 Shape

The model is a normalized map of nodes plus a root reference. Containers reference children by id. Normalization (vs. a nested tree) makes partial updates, diffing, and the wire-format for server-built models trivial:

type NodeId = string;
/** JSON Pointer into the form data, e.g. "/address/street", "/items/2/name" */
type DataPath = string;

interface PresentationModel {
  version: number;              // monotonically increasing per update
  rootId: NodeId;
  nodes: Record<NodeId, PresentationNode>;
}

Deliberate break: the new layer uses JSON Pointer for data paths instead of 3.x lodash dot-paths. JSON Pointer is standard, unambiguous for keys containing dots, and serializes cleanly across the wire.

4.2 Node taxonomy

4.3 Key types

/** Abstracted value classification — what a renderer needs to pick a widget.
 *  Derived from JSON Schema today, from any other source tomorrow. */
type ValueType =
  | 'string' | 'number' | 'integer' | 'boolean'
  | 'date' | 'time' | 'date-time'
  | 'object' | 'array' | 'unknown';

/** Widget-relevant constraints, abstracted away from any schema dialect. */
interface ControlConstraints {
  maxLength?: number;  minLength?: number;
  minimum?: number;    maximum?: number;   multipleOf?: number;
  pattern?: string;
  format?: string;          // escape hatch for source-specific formats
  multiline?: boolean;
}

interface EnumOption {
  value: unknown;
  label: string;            // final, translated
}

/** Validation result attached to nodes — replaces AJV's ErrorObject as public API. */
interface FormIssue {
  severity: 'error' | 'warning' | 'info';
  message: string;          // final, translated, ready to display
  key?: string;             // stable key (validation keyword / i18n key)
  source?: unknown;         // raw validator payload (e.g. AJV ErrorObject), still JSON
}

interface ArrayItem {
  key: string;              // stable per data element, survives reorder → perfect React key
  child: NodeId;            // root node of the item's sub-form
  label?: string;           // computed child label (master/detail, tables)
}

4.4 Design decisions on the node level

  • The model mirrors the expected visual representation. Array nodes have one item entry per element in the data; recursive schemas terminate naturally because the data terminates — the “endless layout” problem of schema-first generation disappears. Lazy materialization of collapsed subtrees (tabs, accordions) is a planned optimization (§14).
  • uiOptions is pre-merged. The merge(config, uischema.options) dance every renderer does today happens once in the builder.
  • Stable array item keys. The engine assigns a persistent key per array element and keeps it across moves — fixing the long-standing index-as-React-key issue and making reorder animations possible.
  • Interaction state lives on the node too. touched is set by a touch command (blur), so “show errors only after interaction” becomes a build-pipeline policy instead of per-renderer-set logic (cf. the recent Angular showErrorsOnTouch feature on master).
  • annotations is the open extension bag for NodeProcessors to attach adopter-specific derived data without forking node types.

5. The builder pipeline & pluggable services

The builder walks the UI structure (explicit uischema, or one generated by the SchemaSource), consults the services at every step, and emits nodes. It is a pure function of its inputs:

5.1 SchemaSource — abstracting the schema format

The single interface between the builder and “whatever describes the data”. The builder never touches JSON Schema directly; it only sees facets:

interface SchemaSource {
  /** Interpret a uischema scope (or structural path) into abstract facets. */
  describe(scope: string): SchemaFacets | undefined;
  /** Default value for a location — used by add-item, variant selection, init. */
  createDefault(scope: string): unknown;
  /** Generate a default UI structure when no uischema is provided. */
  generateUISchema(): UISchemaElement;
}

interface SchemaFacets {
  valueType: ValueType;
  title?: string;
  description?: string;
  constraints?: ControlConstraints;
  enumOptions?: EnumOption[];          // pre-i18n labels
  required?: string[];                 // for objects: required property names
  variants?: VariantInfo[];            // oneOf / anyOf alternatives
  itemFacets?: SchemaFacets;           // for arrays
  readOnly?: boolean;
  deprecated?: boolean;
  defaultValue?: unknown;
}

/** First implementation, ~wrapping today's util/resolvers.ts logic: */
const source = jsonSchemaSource(schema /* JsonSchema draft-04/-07/2020-12 */);

This is where today's $ref resolution, combinator handling, and draft-dialect quirks are encapsulated — resolved once per build, not per render. A future zodSource(zodSchema) or plainSource(definition) implements the same three methods and the whole stack — builder, testers, renderers — works unchanged.

5.2 Validator — abstracting validation

interface FormValidator {
  validate(data: unknown): ValidationIssue[] | Promise<ValidationIssue[]>;
}

interface ValidationIssue {
  path: DataPath;             // which node(s) the issue belongs to
  severity: 'error' | 'warning' | 'info';
  key?: string;               // e.g. 'required', 'maxLength' — i18n lookup key
  message: string;            // default (untranslated) message
  source?: unknown;           // raw validator object for power users
}

/** Implementations planned / enabled by this interface: */
ajvValidator(schema, ajv?)        // default; wraps today's createAjv + validate
standaloneValidator(compiledFn)   // AJV standalone precompiled — CSP-safe, no eval
noValidation()                    // replaces ValidationMode 'NoValidation'
remoteValidator(transport)        // server round-trip validation
composeValidators(...validators)  // schema validation + custom business rules
  • Async is first-class. validate may return a promise. The engine applies the data change and emits the model immediately; issues arrive as a second delta when validation resolves. Sync validators (AJV) behave exactly like today.
  • additionalErrors generalizes to engine.setExternalIssues(issues) — external issues are merged into the same distribution step.
  • Validation display policy (today's ValidateAndShow/ValidateAndHide + touched-based display) becomes a pipeline policy deciding which issues land on which nodes.

5.3 ConditionEvaluator, i18n, NodeProcessors

/** uischema rules (SHOW/HIDE/ENABLE/DISABLE) — today evaluated via AJV conditions.
 *  Pluggable so rules work without AJV and custom condition languages are possible. */
interface ConditionEvaluator {
  evaluate(rule: Rule, data: unknown, context: BuildContext): boolean;
}

/** i18n — same Translator concept as 3.x, but applied once at build time. */
interface BuildI18n {
  locale?: string;
  t: Translator;
  translateIssue?: (issue: ValidationIssue, t: Translator, node: PresentationNode) => string;
}

/** The cross-cutting customization hook: every node flows through this chain. */
type NodeProcessor = (node: PresentationNode, ctx: BuildContext) => PresentationNode;

// Example: organisation-wide label policy, replacing N renderer overrides:
const upperCaseRequired: NodeProcessor = (node) =>
  node.kind === 'control' && node.required
    ? { ...node, label: node.label.toUpperCase() }
    : node;

6. The FormEngine runtime: commands, updates, deltas

The FormEngine is the only stateful object. It owns the data and the current model, processes commands, and notifies subscribers with deltas. It replaces actions + coreReducer + middleware of 3.x.

interface FormEngine {
  getModel(): PresentationModel;
  getNode(id: NodeId): PresentationNode | undefined;
  getData(): unknown;
  dispatch(command: FormCommand): void;
  setExternalIssues(issues: ValidationIssue[]): void;
  subscribe(listener: (delta: ModelDelta) => void): Unsubscribe;       // engine-level
  subscribeNode(id: NodeId, listener: () => void): Unsubscribe;        // node-level
}

type FormCommand =
  | { type: 'set-value';      path: DataPath; value: unknown }
  | { type: 'add-item';       path: DataPath; value?: unknown; index?: number }
  | { type: 'remove-items';   path: DataPath; indices: number[] }
  | { type: 'move-item';      path: DataPath; from: number; to: number }
  | { type: 'select-variant'; nodeId: NodeId; variantIndex: number }
  | { type: 'touch';          nodeId: NodeId }
  | { type: 'update-input';   data?: unknown; uischema?: UISchemaElement }
  | { type: 'update-context'; locale?: string; readonly?: boolean; config?: object };

/** Every command may carry a `sourceNodeId` — the node on whose behalf it was
 *  dispatched (provenance for interceptors, auditing, remote backends). */

/** What subscribers receive — also the wire format for server-driven forms. */
interface ModelDelta {
  version: number;
  upserted: PresentationNode[];   // new or changed nodes
  removed: NodeId[];
  rootChanged?: boolean;
  data?: unknown;                 // optional data echo for onChange consumers
}

const engine = createFormEngine({
  // JSON Schema convenience preset:
  schema, uischema, data,
  // …which is sugar for the fully explicit form:
  schemaSource: jsonSchemaSource(schema),
  validator: ajvValidator(schema),
  conditions: ajvConditionEvaluator(),
  i18n, config, processors: [],
});

Commands replace handleChange + action creators. They are plain JSON — serializable by design, so the same command stream works against a local engine, a web worker, or a server (§8). Today's middleware concept maps to a command interceptor: (command, next) => void.

7. Incremental updates & identity preservation

Issue #2571 calls out that real performance gains need partial updates, not full re-generation. We stage this:

Phase 1 (MVP): full rebuild with structural sharing

  • On every command the builder re-walks the structure, but for each node it compares the node's inputs (value slice identity, issues for its path, rule outcome, label inputs) against the previous build and reuses the previous node object when nothing changed.
  • Cost is O(form size) per change — same asymptotics as 3.x's mapper re-runs, but the work per node is cheaper (no schema resolution: facets are cached per scope) and, crucially, view re-renders drop to O(changed nodes).
  • This is simple, correct, and already a measurable win. It also defines the contract (identity preservation, deltas) that Phase 2 must honor.

Phase 2: dependency-tracked partial updates

  • During the build, each node records its dependencies: its data path prefix, rule condition paths, and label/i18n inputs. The engine maintains an inverted index DataPath → NodeId[].
  • A set-value then recomputes: the nodes covering the changed path, the nodes whose rule conditions reference it, and the nodes whose issue set changed (diffing the validator output between runs).
  • Structural commands (add-item, select-variant) rebuild only the affected subtree.

Risk watch: whole-form validators (AJV) always produce a full issue list, so issue diffing, not issue computation, is the partial-update boundary for validation. Cross-field rules are the other global dependency. The dependency index handles both, but this is the part of the design with the most hidden complexity — hence Phase 2, with Phase 1's full-rebuild as the always-correct fallback path.

8. Deployment modes incl. server-side building

Because the model is serializable JSON, commands are serializable JSON, and deltas are serializable JSON, the engine can sit on either side of a network boundary. The renderer/ binding layer is identical in all three modes — it only ever sees a FormEngine-shaped object:

ModeWhat ships to the browserTypical use
A — Client-only Engine + JSON Schema source + AJV Default; drop-in successor of today's setup.
B — Server-built, client-updated Engine, hydrated from a server-built model SSR/fast first paint; heavy schema preprocessing offloaded.
C — Server-authoritative Only the thin RemoteFormEngine proxy + renderers CSP-restricted environments, secret schemas/business rules, non-JS validation backends, minimal bundles.

For Mode C, the RemoteFormEngine implements the same FormEngine interface over a pluggable transport (HTTP, WebSocket). It echoes set-value into the local node optimistically so typing stays latency-free, and reconciles when the authoritative delta arrives. Transport, batching, and conflict policy are deliberately out of core scope — core defines the protocol types (FormCommand, ModelDelta) only.

9. React binding

The binding shrinks dramatically: no HOC chains, no context-wide cascades, no mapper calls. Three primitives:

// 1. Provider — owns a local engine (convenience) or receives any FormEngine.
<JsonForms
  schema={schema} uischema={uischema} data={data}    // convenience: local engine
  renderers={materialRenderers}
  onChange={({ data, issues }) => ...}
/>
// or:
<JsonForms engine={remoteEngine} renderers={materialRenderers} />

// 2. useNode — node-granular subscription via useSyncExternalStore.
const useNode = (id: NodeId): PresentationNode => {
  const { engine } = useFormContext();
  return useSyncExternalStore(
    (cb) => engine.subscribeNode(id, cb),
    () => engine.getNode(id),
    () => engine.getNode(id)        // SSR snapshot — Mode B works out of the box
  );
};

// 3. NodeDispatch — renderer selection, replacing JsonFormsDispatch.
const NodeDispatch = ({ id }: { id: NodeId }) => {
  const node = useNode(id);
  const { renderers } = useFormContext();
  const match = useMemo(
    () => maxBy(renderers, (r) => r.tester(node)),     // testers see the NODE only
    [renderers, node]
  );
  if (!node?.visible) return null;
  if (!match || match.tester(node) === NOT_APPLICABLE) return <UnknownNode node={node} />;
  return <match.renderer id={id} />;
};

Testers run against presentation nodes, not (uischema, schema) pairs. They become trivial, fast (no schema resolution at match time), and schema-format independent:

type NodeTester = (node: PresentationNode) => number;   // NOT_APPLICABLE = -1

const isStringControl = (n: PresentationNode) =>
  n.kind === 'control' && n.valueType === 'string' && !n.options;

const materialTextControlTester = rankWith(1, isStringControl);

Because testers only need the node, renderer selection can even happen against a serialized, server-built model — and the existing combinators (rankWith, and, or) carry over with a changed signature.

10. Impact on material renderers

Renderers keep their local UI state (focus, dialogs, debounced input) and their MUI markup — they lose all derivation logic. Before/after for the text control:

3.x — logic mixed into the view path

// HOC computes ~20 props via
// mapStateToControlProps on every render
export default withJsonFormsControlProps(
  MaterialTextControl
);

// MaterialInputControl re-derives:
//  - merge({}, config, uischema.options)
//  - isDescriptionHidden(...)
//  - showAsRequired(...)
//  - error string formatting
// MuiInputText reads schema.maxLength,
// uischema options, config again…

4.x — thin view over a ControlNode

const MaterialTextControl = ({ id }) => {
  const node = useControlNode(id);
  const dispatch = useFormDispatch();
  const [text, onText] = useDebounced(
    node.value, (v) =>
      dispatch(setValue(node.path, v)));

  return (
    <TextField
      label={node.label}
      value={text ?? ''}
      required={node.required}
      disabled={!node.enabled}
      error={node.issues.length > 0}
      helperText={node.issues[0]?.message
        ?? node.description}
      inputProps={{ maxLength:
        node.constraints.maxLength }}
      onChange={(e) => onText(e.target.value)}
      onBlur={() => dispatch(touch(id))}
    />
  );
};

Consequences across the renderer set:

  • Layouts map node.children to <NodeDispatch id={childId} /> — no more re-passing schema/path down the tree.
  • Array renderers read node.items (with stable item.key as React key and precomputed item.label) and dispatch add-item/remove-items/move-item. Today's childErrors plumbing disappears — each item's nodes carry their own issues.
  • oneOf/anyOf renderers read node.variants + node.selectedIndex and dispatch select-variant; default-value creation and tab-switch data handling move into the engine where they belong.
  • The cells registry can likely merge into the renderers registry: a table cell is just a compact renderer for a ControlNode in an array context (open question in §14).

12. Repository strategy: staying mergeable with master

The iteration will run for months while master keeps moving. The strategy is strictly additive code in new directories — the 3.x stack is not modified until the very end:

packages/core/src/presentation/        ← all new core code (model, pipeline, engine,
                                          json-schema source, ajv validator, testers)
packages/core/test/presentation/       ← new tests
packages/react/src/presentation/       ← new binding (provider, useNode, NodeDispatch)
packages/material-renderers/src/next/  ← ported renderers, old ones untouched
packages/examples/src/presentation/    ← side-by-side example entries
docs/presentation-model/               ← this document, ADRs, protocol specs
  • Touch existing files only for exports (one export * line in each index.ts, or — preferred — a dedicated subpath export @jsonforms/core/presentation so 3.x consumers see zero surface change and tree-shaking is trivial).
  • Merge master into feat/presentation-model regularly (e.g. weekly). With disjoint directories, conflicts are limited to lockfile/exports noise.
  • Reuse, don't fork, the stable 3.x internals where they fit: util/resolvers.ts (inside jsonSchemaSource), generators/uischema.ts, models/uischema.ts types, util/validator.ts (inside ajvValidator). Master-side fixes to these flow into the new stack via normal merges for free.
  • Old stack removal is the last commit series of the 4.0 release, after the renderer sets are ported — not a precondition for any earlier milestone.

14. Open questions & risks

#Question / riskCurrent lean
1 Node id stability for moved array items. Path-based ids are readable but change on reorder; key-based ids are stable but opaque. Path-based NodeIds + stable ArrayItem.key for view identity; engine remaps ids on move-item.
2 How much raw source on nodes? A raw-schema escape hatch would bloat serialized models (Modes B/C) and tempt renderers back into schema-coupling. Resolved (June 2026): none. Nodes expose abstracted facets only; anything a renderer legitimately needs is added to the node model or via a NodeProcessor.
3 Lazy subtree materialization. Huge categorized forms shouldn't pay full build cost for invisible tabs. Defer to M4: materialize(nodeId) command + materialized: boolean on container nodes.
4 Does uischema stay first-class? The PM abstracts the data schema — should the UI structure description also become pluggable? Keep today's uischema as the canonical UI structure input for 4.0; it is already schema-format-neutral except for scope semantics, which the SchemaSource interprets.
5 Cells vs. renderers. Can the cell registry merge into the renderer registry now that nodes carry everything? Likely yes — table contexts set an annotation (annotations.compact) that testers can rank on. Decide during M3 (table port).
6 Async validation UX. Two-phase deltas (data echo, then issues) may flicker error states. Issue-retention policy in the engine: keep previous issues marked stale until the new result lands.
7 Partial-update correctness (cross-field rules, whole-form validators). Phase-1 full rebuild stays as the always-correct path; Phase 2 must prove equivalence property-based against it.
8 Compat layer worth it? Re-implementing mapStateToControlProps on top of the PM is feasible (the node is a superset) but drags 3.x semantics into 4.x. Decide at M6 with real migration feedback; not on the critical path.
9 Mode C interaction latency for typing-heavy forms. Optimistic local echo for set-value is mandatory in RemoteFormEngine; everything else tolerates round-trips.
